## 0. TL;DR

Agnes ist ein AI-Entscheidungsunterstützungssystem für den Einkauf in der Supplement-Industrie. Sie erkennt, welche firmenspezifischen Rohstoff-SKUs in Wahrheit derselbe Stoff sind, schlägt Lieferanten-Konsolidierungen vor und prüft jede Empfehlung gegen compliance-relevante Anforderungen der betroffenen Endprodukte – mit vollständigem Evidence-Trail.

**Was Agnes besonders macht:**

- **Multi-Agent-Reasoning statt linearer Pipeline** – drei spezialisierte LLM-Rollen (Vorschlagender, Prüfer, Skeptiker) argumentieren gegeneinander; nur was durch den adversarial review kommt, wird empfohlen.
- **Information Gap Loop** – Agnes sagt nicht nur was sie denkt, sondern explizit was sie bräuchte, um sicherer zu sein. Unsicherheit wird zu einer konkreten Handlungsaufforderung.
- **Scalability-Story über Feedback-Lernen** – jede angenommene/abgelehnte Empfehlung verbessert die Confidence-Kalibrierung über Zeit.

> Das Tool entscheidet nicht. Es berät Einkäufer. Der Mensch bleibt in der Schleife.

## 1. Datenlage – was die Datenbank wirklich hergibt

Wir arbeiten auf einer SQLite-DB mit 6 Tabellen:

| Tabelle | Zeilen | Bemerkung |
|---|---|---|
| Company | 61 | Endmarken (Caltrate, Centrum, Body Fortress, iherb-Brands) |
| Product | 1025 | 149 finished-goods + 876 raw-materials |
| BOM | 149 | Ein BOM pro finished-good |
| BOM_Component | 1528 | M:N zwischen BOMs und Rohstoffen |
| Supplier | 40 | 40 Lieferanten |
| Supplier_Product | 1633 | M:N zwischen Lieferanten und Rohstoffen |

**Was vorhanden ist**
- Welche Firma verkauft welches Endprodukt
- Welche Rohstoffe gehen in welches Endprodukt
- Welcher Lieferant beliefert welchen Rohstoff
- Substanznamen sind im SKU-String kodiert: `RM-C1-calcium-citrate-05c28cc3`

**Was fehlt (und extern beschafft werden muss)**
- Preise, Lead Times, MOQs, Länder
- Zertifizierungen, Compliance-Felder, Spezifikationen
- Mengen pro BOM-Komponente
- Historische Preisentwicklung

> **Konsequenz:** Unser gesamter Mehrwert hängt an der Kombination aus interner Fragmentierungs-Analyse und externer Evidenz-Beschaffung. Wer nur die DB nutzt, baut einen SKU-Matcher. Wer extern anreichert, baut ein Entscheidungssystem.

**Fragmentierungs-Beobachtung:**
Rohstoffe sind pro Firma ge-SKU't. `RM-C1-calcium-citrate-...` gehört Company 1, `RM-C14-calcium-citrate-...` gehört Company 14. Derselbe Stoff existiert also vermutlich 10–30 Mal mit unterschiedlichen IDs. Genau diese Fragmentierung ist das Problem, das Agnes löst.

## 2. Architektur

### 2.1 Skeleton – 5 Schichten

```
┌────────────────────────────────────────────────────────────┐
│ SCHICHT 1 · DATEN                                          │
│   SQLite DB (Companies, Products, BOMs, Suppliers, M:N)    │
└────────────────────────────────────────────────────────────┘
                           ↓
┌────────────────────────────────────────────────────────────┐
│ SCHICHT 2 · VORVERARBEITUNG                                │
│   SKU-Normalisierung → Substanz aus Namen extrahieren      │
│   Embedding-Clustering → Kandidaten-Gruppen bilden         │
│   Fragmentierungs-Analyse → wer kauft was bei wem?         │
└────────────────────────────────────────────────────────────┘
                           ↓
┌────────────────────────────────────────────────────────────┐
│ SCHICHT 3 · MULTI-AGENT REASONING                          │
│   🟠 Equivalence-Agent  →  🟢 Compliance-Agent             │
│                          →  🔴 Devil's Advocate            │
│                          →  🟣 Arbiter (deterministisch)   │
└────────────────────────────────────────────────────────────┘
                           ↓       ↕ (Evidenz)
┌────────────────────────────────────────────────────────────┐
│ SCHICHT 4 · EXTERNAL ENRICHMENT                            │
│   Web-Scraping · Zertifikats-Lookup · Evidence-Cache       │
└────────────────────────────────────────────────────────────┘
                           ↓
┌────────────────────────────────────────────────────────────┐
│ SCHICHT 5 · OUTPUT                                         │
│   Empfehlungs-Memo · 💡 Information Gap · Einkäufer-UI     │
└────────────────────────────────────────────────────────────┘
                           ↑
                   Feedback-Loop
```

### 2.2 Multi-Agent-Reasoning im Detail

Alle Agenten sind derselbe LLM, aber mit unterschiedlichen Rollen-Prompts. Sie kommunizieren über einen festen JSON-Kontrakt.

| Agent | Rolle | Zentrale Frage |
|---|---|---|
| 🟠 Equivalence | Einkäufer | Welche SKUs sind funktional austauschbar, wie konsolidieren? |
| 🟢 Compliance | QM / Compliance-Officer | Erfüllt der Lieferant die Compliance-Anforderungen jedes Endprodukts? |
| 🔴 Devil's Advocate | Skeptiker | Welche Risiken werden übersehen? Was bricht den Vorschlag? |
| 🟣 Arbiter | Regel-Gate (kein LLM) | Deterministische Entscheidung aus allen Outputs |

> **Warum der Arbiter kein LLM ist:** Die finale Entscheidung muss reproduzierbar sein. LLMs sind für Reasoning da, nicht für finale Gates. Eine deterministische Regel („ein Einwand mit Schweregrad 'hoch' → blockiert") ist erklärbar, testbar und robust.

### 2.3 Information Gap Loop

Das Feature, das uns vom Rest abhebt. Jeder Agent meldet nicht nur sein Urteil, sondern auch was er nicht verifizieren konnte.

```
Equivalence-Agent    ─┐
Compliance-Agent     ─┤
Devil's Advocate     ─┤→ Gap-Aggregator → Gap-Ranker → Information Request
                      │                                        ↓
                      └←←←  User liefert fehlende Info    ←   Einkäufer
```

**Output-Beispiel:** „Um die Confidence von 62% auf 91% zu heben, brauche ich: (1) Halal-Zertifikat für ADM, (2) Food-Grade-Spec für SKU-42, (3) 2025er Allergen-Report."

## 3. Technischer Stack

| Layer | Technologie | Begründung |
|---|---|---|
| Sprache | Python 3.11 | Data-Science-Standard, alle haben Erfahrung |
| DB-Zugriff | SQLAlchemy (laut Briefing) | ist mitgegeben |
| Embeddings | OpenAI text-embedding-3-small oder sentence-transformers | beides ok, Embeddings sind billig |
| LLM | Claude / GPT-4 / lokales Modell | egal – nur die Prompts müssen stabil sein |
| Clustering | scikit-learn (AgglomerativeClustering) | einfach, deterministisch, erklärbar |
| Web-Enrichment | requests + beautifulsoup4 + LLM-Extraktion | eine Quelle reicht für MVP |
| Orchestrierung | Simple Python-Funktionsaufrufe | kein LangGraph o.ä. – KISS |
| UI | Streamlit | schneller Prototyp, keine Frontend-Zeit |
| Caching | diskcache oder JSON-Files | Demo muss offline laufen |

> **Kein LangChain, kein LangGraph, kein Vektor-DB-Overkill.** Die Komplexität liegt in den Prompts und im Reasoning, nicht im Framework.

## 4. JSON-Kontrakt zwischen Agenten

> Dieser Kontrakt ist das erste Artefakt, das steht. **Vor Code.**

### 4.1 Input an Equivalence-Agent

```json
{
  "cluster_id": "calcium-citrate-001",
  "skus": [
    "RM-C1-calcium-citrate-05c28cc3",
    "RM-C14-calcium-citrate-...",
    "RM-C27-calcium-citrate-..."
  ],
  "affected_companies": ["Caltrate", "Centrum", "Body Fortress"],
  "affected_boms": [12, 47, 83],
  "current_suppliers": ["ADM", "Cargill", "BulkSupplements"],
  "end_product_context": [
    { "sku": "FG-iherb-10421", "company": "Caltrate", "name_hint": "..." }
  ]
}
```

### 4.2 Einheitliches Agent-Output-Schema

```json
{
  "agent": "equivalence",
  "verdict": "PROPOSED",
  "confidence": 0.74,
  "reasoning": "Kurze Natural-Language-Begründung",
  "claims": [
    {
      "claim": "SKU A und B sind chemisch identisch",
      "evidence_url": "https://...",
      "verified": true
    }
  ],
  "objections": [
    { "issue": "Halal-Zertifikat fehlt", "severity": "high" }
  ],
  "missing_evidence": [
    {
      "what": "Halal-Zertifikat für ADM Calciumcitrat",
      "why_it_matters": "4 der 6 Endprodukte sind halal-zertifiziert",
      "confidence_lift_if_provided": 0.18
    }
  ]
}
```

### 4.3 Verdict-Werte (deterministische Arbiter-Regel)

| Agent-Outputs | Arbiter-Verdict |
|---|---|
| Keine Objections, Confidence ≥ 0.7 | `RECOMMENDED` |
| Objections nur low, Confidence ≥ 0.6 | `RECOMMENDED_WITH_CAVEAT` |
| Mind. eine Objection medium | `REVIEW_REQUIRED` |
| Mind. eine Objection high | `BLOCKED_PENDING_EVIDENCE` |

## 5. Agent-Prompts (Skelett)

### 5.1 Equivalence-Agent

```
Du bist ein erfahrener Einkäufer in der Supplement-Industrie.
Ziel: Konsolidierungs-Potenzial im gegebenen Cluster identifizieren.

Input: Ein Cluster von SKUs aus verschiedenen Firmen, die potenziell
denselben Rohstoff bezeichnen, inklusive aktueller Lieferanten und
betroffener Endprodukte.

Deine Aufgabe:
1. Bestätige, dass die SKUs tatsächlich denselben Stoff bezeichnen,
   oder flagge Abweichungen (z.B. unterschiedliche Grades).
2. Schlage einen konsolidierten Lieferanten vor.
3. Begründe kurz (2-3 Sätze).

Antworte AUSSCHLIESSLICH im vorgegebenen JSON-Schema.
Setze "missing_evidence" aktiv, wenn dir Info fehlt.
Erfinde niemals Evidenz-URLs. Lieber "missing_evidence" setzen
als einen nicht-existenten Claim zu konstruieren.
```

### 5.2 Compliance-Agent

```
Du bist ein Compliance-Officer für Nahrungsergänzungsmittel.

Input: Ein Konsolidierungs-Vorschlag vom Equivalence-Agent plus
die Liste der betroffenen Endprodukte.

Deine Aufgabe:
1. Leite aus den Endprodukten die Compliance-Anforderungen ab
   (z.B. halal, kosher, vegan, non-GMO, allergenfrei).
2. Prüfe anhand der bereitgestellten externen Evidenz,
   ob der vorgeschlagene Lieferant diese erfüllt.
3. Markiere offene Lücken explizit in "missing_evidence".

Du darfst nur Claims mit "verified": true markieren, wenn du eine
konkrete Evidence-URL dafür hast. Alles andere ist "verified": false.

Antworte AUSSCHLIESSLICH im vorgegebenen JSON-Schema.
```

### 5.3 Devil's Advocate

```
Du bist ein gnadenloser Skeptiker. Dein einziger Job ist es,
Gründe zu finden, warum dieser Vorschlag NICHT umgesetzt werden sollte.

Input: Ein Konsolidierungs-Vorschlag plus Compliance-Bericht.

Deine Aufgabe:
- Finde übersehene Risiken: Single-Sourcing, Supply-Chain-Abhängigkeit,
  fehlende Zertifikate, regulatorische Änderungen, Saisonalität,
  Qualitäts-Drift zwischen Lieferanten, etc.
- Jede Objection bekommt einen Schweregrad: low / medium / high.
- Sei spezifisch. "Supply-Chain-Risiko" ohne Begründung ist wertlos.

Wenn du keine Objection findest, gib eine leere Liste zurück -
aber nicht ohne ernsthaft nach Problemen gesucht zu haben.

Antworte AUSSCHLIESSLICH im vorgegebenen JSON-Schema.
```

### 5.4 Gap-Ranker (kleiner zusätzlicher LLM-Call)

```
Input: Alle missing_evidence-Einträge aus den drei Agenten.

Deine Aufgabe: Priorisiere sie nach:
1. confidence_lift_if_provided (absteigend)
2. Wie einfach die Info beschaffbar ist (Einschätzung)
3. Wie blockierend sie für die Freigabe ist

Gib die Top-3 zurück, formatiert als konkrete Handlungsanweisung
für den Einkäufer: "Klär X, dann Y, dann Z."
```

## 8. Demo-Storyline

> Der rote Faden der 5-Minuten-Präsentation.

**Akt 1 · Das Problem (60s)**
„In der Supplement-Industrie kauft jede Firma ihre Rohstoffe unter eigenen SKU-Namen ein. 61 Firmen, 876 Rohstoff-SKUs, 40 Lieferanten. Niemand sieht, dass Calciumcitrat in 18 Produkten bei 8 verschiedenen Lieferanten beschafft wird – obwohl es chemisch derselbe Stoff ist. Ergebnis: schlechtere Preise, keine Verhandlungsmacht."

**Akt 2 · Die naive Lösung (30s)**
„Man könnte einen LLM fragen: 'Sind diese zwei SKUs austauschbar?'. Das Problem: der LLM ist zu nett. Er sagt fast immer ja."

**Akt 3 · Agnes (90s) – das Herzstück**
„Wir lassen drei Agenten streiten." Live-Demo mit Calciumcitrat:

1. **Equivalence-Agent:** „Alle 18 SKUs sind derselbe Stoff, konsolidieren auf ADM."
2. **Compliance-Agent:** „ADM hat Kosher, Non-GMO – aber Halal-Zertifikat fehlt öffentlich, 4 Endprodukte sind halal."
3. **Devil's Advocate:** „Ohne Halal blockiert. Außerdem: Single-Sourcing-Risiko mittel."
4. **Arbiter:** `BLOCKED_PENDING_EVIDENCE`

**Akt 4 · Information Gap (60s) – der Jury-Moment**
„Agnes blockiert nicht einfach. Sie sagt dem Einkäufer exakt, was er prüfen muss, um die Freigabe zu ermöglichen: aktuelles Halal-Zert für ADM, benannter Backup-Lieferant. Zwei konkrete Tickets, die ein Einkäufer in 2 Tagen erledigen kann."

**Akt 5 · Skalierung (30s)**
„Jede Entscheidung, die ein Einkäufer trifft, trainiert die Confidence-Kalibrierung. Nach 50 Entscheidungen weiß Agnes, wo sie historisch richtig lag. Nach 500 kennt sie die Compliance-Präferenzen jeder Firma. Das ist der Unterschied zwischen einem Modell und einem Produkt."

**Akt 6 · Schluss (30s)**
„Agnes entscheidet nicht. Sie macht den Einkäufer schneller und sicherer. Und sie ist ehrlich dazu, was sie nicht weiß – das ist ein seltenes Feature für ein KI-System."

> Total: 5 Minuten. Wir bauen die ganze Demo um Akt 3 und 4 herum.

## 10. Risiken & Gegenmaßnahmen

| Risiko | Gegenmaßnahme |
|---|---|
| LLM halluziniert Compliance-Belege | Prompt zwingt zu `evidence_url` ODER `missing_evidence`. Kein Claim ohne URL. |
| LLM gibt kein valides JSON zurück | Retry mit Fehler-Meldung. Fallback: Parsing-Heuristik. OpenAI `response_format={"type":"json_object"}`. |
| External Enrichment scheitert live | Alle Scrapes während Entwicklung cachen. Demo läuft offline. |
| Clustering gruppiert zu grob/fein | LLM-Verifier als zweite Stufe. Schwellwert konfigurierbar. |
| Scope-Kriechen Samstag Nachmittag | Sa 22:00 Feature-Freeze. Wer dagegen verstößt, kostet dem Team die Demo. |
| Demo crasht vor Jury | Screen-Recording als Backup. Generalprobe Sonntag 09:00. |
| Team arbeitet durch und ist k.o. | Zwei feste Schlafblöcke geplant, Wecker gestellt. |
| Devil's Advocate findet nie was | Few-shot examples im Prompt mit echten Risiken. |
| Arbiter-Regel zu strikt, immer BLOCKED | Threshold konfigurierbar, für Demo anpassbar. |

## 16. Anhang · Beispiel-Queries auf der DB

```sql
-- Welche Substanzen kommen in den meisten BOMs vor?
SELECT
  substr(SKU, instr(SKU, '-', 5) + 1,
         length(SKU) - instr(SKU, '-', 5) - 10) AS substance,
  COUNT(DISTINCT bc.BOMId) AS bom_count
FROM Product p
JOIN BOM_Component bc ON p.Id = bc.ConsumedProductId
WHERE p.Type = 'raw-material'
GROUP BY substance
ORDER BY bom_count DESC
LIMIT 20;

-- Fragmentierung: wie viele verschiedene SKUs pro Substanz?
SELECT substance, COUNT(*) AS sku_count
FROM (...)
GROUP BY substance
HAVING sku_count > 5;

-- Welche Substanzen haben die meisten Lieferanten?
SELECT substance, COUNT(DISTINCT sp.SupplierId) AS supplier_count
FROM ... JOIN Supplier_Product sp ...
GROUP BY substance
ORDER BY supplier_count DESC;
```
